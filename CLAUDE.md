## Iron Rules (from Superpowers + ECC RULES.md)

These are NON-NEGOTIABLE. Violating them is grounds for reverting the change.

1. **NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST.** (systematic-debugging)
   If you haven't reproduced the bug and read the error message fully, you
   cannot propose a fix. Symptom fixes are failure.
2. **NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE.**
   (verification-before-completion)
   Don't say "tests pass" / "build succeeds" / "bug fixed" without running the
   command in the current message and showing its output. "Should pass" is a lie.
3. **NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST.** (test-driven-development)
   For new features + bugfixes: write test → watch it fail → write minimal code.
   Exceptions: throwaway prototypes, generated code, config files.
4. **NEVER COMMIT SECRETS.** No API keys, tokens, passwords, .env contents in
   output. ECC RULES.md "Must Never" #1.
5. **NEVER BYPASS SECURITY CHECKS.** ECC RULES.md "Must Never" #3.
   If a hook/validator blocks you, fix the input — don't disable the hook.

## How to Work in This Repo

### Before writing code

1. **Read existing patterns first.** ECC RULES.md "Must Always" #5: "Follow
   established repository patterns before inventing new ones." Check how a
   similar feature is implemented (e.g. look at `process_video_job` before
   writing `process_music_job`).
2. **Search before adding.** ECC "Must Never" #4: don't duplicate functionality.
3. **Plan before implementing.** Use the `Plan` subagent for design exploration.

### When debugging

Apply Superpowers `systematic-debugging`:
1. Phase 1: Root Cause Investigation — read errors, reproduce, check `git diff`,
   add diagnostic instrumentation at component boundaries.
2. Phase 2: Hypothesis Formation — single most likely cause, written down.
3. Phase 3: Fix Implementation — minimal change addressing root cause.
4. Phase 4: Verification — re-run reproduction, confirm fix, no regressions.

### When claiming completion

Apply Superpowers `verification-before-completion`:
| Claim | Required evidence |
|---|---|
| "Tests pass" | `pytest tests/` output: 0 failures (in THIS message) |
| "Build succeeds" | `npm run build` / `docker-compose build` exit 0 |
| "Bug fixed" | Reproduction script: passes |
| "Lint clean" | `ruff check .` / `tsc --noEmit`: 0 errors |

Skip any step = lying, not verifying.

### When delegating to subagents

Apply ECC `agent-format` + claude-code-best-practice `claude-subagents.md`:
- Pass clear, self-contained task descriptions (subagents can't see your context).
- Specify `model: haiku` for fast searches, `sonnet` for analysis, `opus` for design.
- Read-only subagents (`Explore`) for audit/investigation; never let a research
  subagent write code (it lacks full context).
- After subagent returns, verify its claims against the actual files (don't trust
  the report — read the diff).

## Common Commands

```bash
# Python validation (run after ANY .py edit)
python -m py_compile <file>              # syntax check one file
python -c "import ast; ast.parse(open('<file>').read())"  # AST parse
find . -name '*.py' -not -path '*/.venv/*' | xargs -n1 python -m py_compile  # all

cd miniapp && npx tsc --noEmit           # typecheck miniapp

# YAML validation

# Tests
pytest tests/ -x --tb=short              # stop on first failure
pytest tests/test_payments.py -v         # single domain

# Docker

docker-compose up -d                     # start all
docker-compose logs -f api bot worker    # tail logs

# Migrations
alembic upgrade head                     # apply all pending
alembic downgrade -1                     # rollback one
alembic revision -m "describe change"    # autogenerate
```
